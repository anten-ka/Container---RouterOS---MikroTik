# Хендофф-документ: проект «MikroTik + VLESS для подписчиков»

> Документ для передачи другому ИИ-агенту на ревью и правку. Ниже — цель проекта, полная хронология обсуждения, что сделано, ключевые технические факты, какие решения приняты и почему, что протестировано и — главное — **список мест, которые стоит перепроверить** (с приоритетами).

---

## 1. Цель проекта

Пользователь (Виталий) распространяет платную инструкцию **«№30 — Mikrotik + VLESS (+ маршрутизация)»** — как поднять на роутере MikroTik VLESS-туннель через Docker-контейнер, с частичной маршрутизацией (часть трафика в VPN, часть напрямую).

Две задачи, которые решались по ходу диалога:

1. **Диагностика.** Разобраться, почему у пользователей «не ставится / не настраивается контейнер», и дать точечные фиксы. Частая жалоба — «не скачивается».
2. **Инструмент «в один клик».** Сделать максимально удобный для конечного пользователя способ настройки: чтобы не вводить руками десятки команд, а ответить на вопросы и получить готовый результат. В идеале — чтобы всё поставилось само.

Конечная аудитория — **нетехнические люди** (подписчики автора). Отсюда требование к максимальной автоматизации и «защите от дурака».

## 2. Технический контекст

- **Исходник инструкции:** Google Doc автора (Telegram-канал `@antenkaru`). Инструкция №30; есть более простой аналог №32 (на ядре Mihomo).
- **Контейнер:** `wiktorbgu/vless-sing-box-tunnel-mikrotik` (Docker Hub). Автор контейнера — chat `@it_network_people`.
  - Образ **мультиархитектурный**: `arm/v7`, `arm64`, `amd64`. Живой, ~17 МБ сжатый. Проблема почти никогда не в самом образе.
  - **Важно:** автор контейнера пометил sing-box-вариант как устаревший и рекомендует свой новый `wiktorbgu/mihomo-mikrotik`.
- **Пример железа:** MikroTik hAP ax² — это **arm64**. Требования: RouterOS 7.5+, ARM, ≥128 МБ ROM, поддержка контейнеров.
- **Топологии в инструкции:**
  - *Прямая* — 1 зарубежный VPS.
  - *Каскадная (двойной VLESS)* — Москва VPS → зарубежный VPS (рекомендуется автором).
- **Смежные компоненты инструкции:** панель 3x-ui (получение ключей), скрипт WhiteVPN (`blockme`), сборка списков маршрутов через **DomainMapper** (репозиторий `Ground-Zerro/DomainMapper`).
- **Схема сети контейнера (из инструкции):** veth `VLESS-TUN-SBOX`, адреса `172.18.20.5/30` (роутер) и `172.18.20.6/30` (шлюз в контейнер), MSS-клампинг, masquerade, переменные окружения из ключа `vless://`.

## 3. Хронология: что обсуждали и делали

1. **Запрос 1 — «проверь по инструкции, почему может не работать установка контейнера».**
   Забрал текст инструкции из Google Doc, дока контейнера с Docker Hub, официальную доку MikroTik (Container, Disks), проверил теги образа через API Docker Hub. Уточнил у пользователя, на каком шаге падает → ответ: «нужно проанализировать все ситуации, но часто говорят не скачивается».
   → Сделал **памятку по диагностике** (файл `mikrotik-vless-container-troubleshooting.md`).

2. **Запрос 2 — «сделай скрипт, который вводишь один раз и он всё настроит».**
   → Сделал статический **`vless-mikrotik-autosetup.rsc`** (заполнить переменные из ключа → `/import`). Скрипт проверяет пакет и device-mode, чинит сертификаты/время, создаёт veth/firewall/tmpfs, envs (авто-выбор синтаксиса `list=`/`name=`), качает контейнер, ждёт распаковку, запускает.

3. **Запрос 3 — «а можно пошаговый скрипт, который сам задаёт вопросы (частичная маршрутизация или отдельная сеть, ключ или подписка) и раскладывает по зависимостям?»**
   Пояснил: **язык скриптов RouterOS не умеет интерактивный ввод** (скрипт не может остановиться и спросить). Поэтому сделал **HTML-мастер `vless-mikrotik-wizard.html`** — задаёт вопросы в браузере на ПК и генерирует готовый `.rsc`. Парсит `vless://` и base64-подписки, режимы маршрутизации (частичная/весь трафик/без).

4. **Запрос 4 — «встрой DomainMapper прямо в скрипт, чтобы он сам собрал список».**
   → Встроил мини-DomainMapper в браузере: тянет списки доменов из репозитория `Ground-Zerro/DomainMapper`, резолвит через **DNS-over-HTTPS** (Cloudflare/Google), агрегирует в подсети (/16, /24, /32), склеивает и кладёт в поле маршрутов. Прогресс-бар, отмена, опция добавить сети CDN Cloudflare.

5. **Запрос 5 (большой пакет) —** добавить:
   - отдельную **Wi-Fi сеть + подсеть**, где живёт VPN;
   - **авто-режим**: указать логин/пароль/IP роутера, чтобы «само зашло и ввело команды», включая ручные предусловия (npk, device-mode); при неудаче — переход в ручной режим;
   - **проверку архитектуры и свободного места**, выбор установки на внутреннюю память или **USB**;
   - **загрузку своего списка доменов**;
   - «скрытые» названия Facebook/Instagram с **дисклеймером про Meta** и обязательной галочкой соответствия (не для РФ);
   - **убрать порт 8443** из подсказки;
   - учесть **лимит на количество вводимых в терминал строк** — применять батчами.

   → Всё реализовано в мастере (детали в разделе 4). Авто-режим — через генерацию **PowerShell-скрипта `setup.ps1`** (браузер по SSH ходить не может), который сам ставит npk, включает device-mode (просит один раз дёрнуть питание), заливает `.rsc` по scp и делает `/import`; при сбое печатает ручную инструкцию.

6. **Запрос 6 — этот документ (хендофф на ревью).**

## 4. Итоговые артефакты (3 файла)

### 4.1 `mikrotik-vless-container-troubleshooting.md`
Памятка по диагностике: таблица «симптом → причина → фикс», разбор «не скачивается» (8 причин по порядку), «не стартует», «running но нет интернета», отличия инструкции от доки автора, чек-лист команд для сбора диагностики. Здесь же зафиксированы ключевые находки (см. раздел 5).

### 4.2 `vless-mikrotik-autosetup.rsc`
Статический скрипт для тех, кто хочет просто заполнить переменные и запустить. По сути — предшественник мастера; логика та же, но без Wi-Fi/USB/PowerShell/батчинга. Оставлен как «простой вариант».

### 4.3 `vless-mikrotik-wizard.html` — основной инструмент
Автономный HTML-файл (чистый клиент, без бэкенда; ничего не отправляет наружу, кроме DoH-резолвинга и загрузки списков доменов с GitHub). Шаги:

1. **Ключ/подписка** — парсит `vless://`, несколько ключей, или base64-подписку; раскладывает на 8 env-переменных (pbk→PUBLIC_KEY, sid→SHORT_ID, sni→SERVER_NAME и т.д.), предупреждает о подозрительном (не Reality, пустой flow, нет pbk).
2. **Маршрутизация** — частичная (список подсетей) / весь трафик (с маршрутом-исключением до VPS от петли) / без маршрутов. Встроенный DomainMapper (DoH), загрузка своего списка доменов файлом или вставкой.
3. **Отдельная Wi-Fi для VPN** — второй SSID + подсеть /24 + DHCP + policy-routing (kill-switch: если туннель упал — в этой сети нет интернета).
4. **Способ установки** — авто по SSH (`setup.ps1`) либо вручную (копипаст/`import`).
5. **Дополнительно** — внутренняя память/USB, логин Docker Hub, имя veth и root-dir.
6. Галочка соответствия (обязательна) → генерация `.rsc` (+ `setup.ps1` в авто-режиме).

**Что генерируемый `.rsc` делает по шагам:** проверки (arch ∈ arm/arm64/x86 — иначе стоп; свободное место — жёсткий стоп <30 МБ, предупреждение <50; пакет container; device-mode) → сертификаты/NTP → хранилище (внутренняя/USB-автопоиск) + registry/tmpdir → veth+адресация → firewall MSS(1360)+masquerade → envs (авто `list=`/`name=`) → добавление контейнера (`dns=1.1.1.1,8.8.8.8`) → ожидание распаковки → старт → (опц.) Wi-Fi → маршруты **батчами по 30 в одну `:foreach`-строку**.

**PowerShell `setup.ps1` (авто-режим):** временный SSH-ключ → определяет версию/arch → при отсутствии пакета качает Extra packages `all_packages-<arch>-<ver>.zip`, распаковывает, шлёт `container-*.npk` по scp, ребутит, ждёт → device-mode=yes + просит физический power-cycle → scp `.rsc` + `/import` → уборка ключа. Любой сбой → функция `Manual` с ручной инструкцией.

## 5. Ключевые технические находки (проверять при ревью контента памятки)

- **hAP ax² = arm64.** В инструкции на скрине фигурирует `container-7.19.3-arm.npk` — на ax² такой (arm, 32-бит) молча не встанет. Это фактическая ошибка инструкции.
- **Два жёстких ручных предусловия** (скриптом нельзя): (1) установка `container-<версия>-<arch>.npk` (Files → reboot); (2) `/system/device-mode/update container=yes` + **отключение питания** (не программный reboot).
- **`tmpdir=ramstorage`** в инструкции задаётся, но диск нигде не создаётся → pull висит/падает. Фикс: создать tmpfs или убрать tmpdir.
- **«Не скачивается»** чаще всего = device-mode не применён / неверный npk / нет tmpdir / лимиты Docker Hub (429) или недоступность `registry-1.docker.io` из РФ. Универсальный обход — импорт образа `.tar` (podman/docker save → scp → `/container/add file=`).
- **Опечатка в инструкции:** `tcp-mss=1453-6553` → должно быть `1453-65535` (в скрипте исправлено).
- **Автор контейнера рекомендует mihomo** вместо sing-box — если у пользователя всё падает, быстрее перейти на инструкцию №32.

## 6. Что протестировано

- Синтаксис JS (`node --check`) — OK.
- Симуляция `generate()` с мок-DOM по комбинациям: manual/auto × int/usb × list/all/none × с Wi-Fi и без; проверка баланса фигурных скобок в RouterOS-выводе (везде 0), срабатывание галочки соответствия и валидации порта.
- Батчинг: 1000 маршрутов → 34 строки `:foreach`, самая длинная строка 565 символов (с запасом под лимит терминала).
- `setup.ps1`: статические проверки квотинга (аргументные массивы, нет `cmd /c`, нет «висячих» кавычек, есть arch-guard, URL npk, промпт power-cycle, fallback Manual).
- Живьём: доступность DoH `dns.google` и `cloudflare-dns.com` (CORS `*`), доступность `platformdb` и списков доменов на raw.githubusercontent (CORS `*`), корректный резолвинг тестовых доменов.
- Баланс HTML-тегов (div/script/style/details/textarea) — OK.

**НЕ тестировалось на реальном железе** (у меня нет доступа к роутеру и к Windows+RouterOS по SSH).

## 7. На что обратить внимание ревьюеру (по приоритету)

### 🔴 Высокий риск — проверить обязательно
1. **Авто-создание Wi-Fi (самое непроверенное).** Зависит от модели/прошивки: `wifiwave2` (`/interface/wifi`) vs старый `/interface/wireless`; имена радиоинтерфейсов; корректность фильтра «мастер-интерфейс» (я определяю через `:do {get master-interface}` — надёжно ли на всех версиях?); создаётся ли виртуальный AP с наследованием диапазона. Проверить на hAP ax² и на устройстве со старым wireless.
2. **PowerShell по SSH.** Никогда не запускался на реальной паре Windows↔RouterOS. Проверить: импорт ключа `/user/ssh-keys/import`, путь scp в домашнюю папку (`user@ip:`), URL и имя файла в архиве Extra packages (`all_packages-<arch>-<ver>.zip` → `container-*.npk`), формат ответа `/system/device-mode/get container` (ожидаю `yes`), поведение при перезагрузке (таймауты), совместимость с Windows PowerShell 5.1 (`-notin`, `Expand-Archive`).
3. **Policy-routing для Wi-Fi kill-switch.** Проверить синтаксис `/routing/table/add name=vpn-rt fib`, `mark-routing`/`new-routing-mark`, `routing-table=` в `/ip/route/add`; убедиться, что трафик самого контейнера к VPS не попадает в vpn-rt и не создаёт петлю; что DNS из подсети (dhcp dns-server=1.1.1.1) реально уходит в туннель.

### 🟡 Средний риск
4. **Различия версий RouterOS.** `envs list=` vs `name=` (обработано try/catch, но проверить); `certificate/settings/set builtin-trust-anchors=trusted` только 7.19+; синтаксис tmpfs-диска; `device-mode/get`.
5. **Автопоиск USB.** Эвристика «первый слот с type≠tmpfs» — не выберет ли случайно системный/неподходящий носитель? Проверить на устройстве с реальным USB.
6. **Пороги свободного места** (30/50 МБ) — прикидка, уточнить под реальный размер распаковки sing-box.
7. **Лимит терминала.** Батч 30/строка и «длинная строка ~565 символов» — подтвердить фактический лимит длины строки и объёма вставки в консоли MikroTik (SSH и Winbox отличаются). При больших списках основной путь — `/import` из файла (в авто-режиме — scp+import), это должно снимать проблему; проверить, что это действительно так.

### 🟢 Ниже риск / содержательное
8. **DoH может резаться провайдером** — предусмотрен второй резолвер; стоит ли добавить ещё вариант (например, свой DoH)?
9. **Docker Hub:** авторизация (`/container/config username/password`) и зеркала — актуальность.
10. **Юридические формулировки** (Meta-дисклеймер, галочка соответствия) — оценить достаточность/корректность.
11. **UX-мелочи:** тексты подсказок, порядок шагов, поведение при частично заполненных полях.

## 8. Открытые вопросы к продукту (не решены — на усмотрение)

- Стоит ли по умолчанию переводить пользователей на **mihomo-контейнер / инструкцию №32**, раз автор считает sing-box устаревшим? Возможно, сделать в мастере выбор ядра.
- Нужна ли поддержка **не-Windows** (macOS/Linux) для авто-режима — сейчас только PowerShell. Можно добавить генерацию `.sh` (bash + ssh/scp).
- Обработка **каскадной топологии** в мастере пока неявная (пользователь просто вставляет ключ от первого сервера). Возможно, стоит явно провести по двум серверам.

## 9. Точные параметры (чтобы ревьюер не искал)

- veth: `VLESS-TUN-SBOX`, роутер `172.18.20.5/30`, шлюз-в-контейнер `172.18.20.6`.
- MSS: `change-mss ... new-mss=1360 ... tcp-mss=1453-65535`.
- NAT: `masquerade ... out-interface=VLESS-TUN-SBOX`.
- Контейнер: `remote-image=wiktorbgu/vless-sing-box-tunnel-mikrotik`, `envlist=vless`, `dns=1.1.1.1,8.8.8.8`, `start-on-boot=yes`.
- 8 env: REMOTE_ADDRESS, REMOTE_PORT, ID, FLOW, FINGER_PRINT, SERVER_NAME, PUBLIC_KEY, SHORT_ID.
- Wi-Fi (по умолчанию): SSID `Home-VPN`, подсеть `192.168.99.0/24`, шлюз `.1`, пул `.10–.254`, таблица `vpn-rt`, метка `vless-vpn-mark`.
- Батч маршрутов: 30 CIDR на строку `:foreach n in={...} do={ /ip/route/add dst-address=$n gateway=172.18.20.6 comment="..." }`.
- Источники списков: `github.com/Ground-Zerro/DomainMapper` (`platformdb` + `platforms/dns-*.txt`). DoH: `dns.google`, `cloudflare-dns.com`.
