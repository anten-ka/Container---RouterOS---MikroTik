## 1. Проверка фактов по документации MikroTik

Проверялись 🔴/🟡 пункты хендоффа против официальной документации.

**🔴 1. Авто-создание Wi-Fi — ПОДТВЕРЖДЁН БАГ.** В новом драйвере wifi (wifi-qcom, hAP ax² и новее) у `/interface/wifi/add` нет параметра `ssid=` — SSID и режим задаются как `configuration.ssid=` и `configuration.mode=ap`, пароль — `security.passphrase=` (док. «Configuring standalone access point», пример виртуального AP). Старый код генерировал `add ... ssid="..."` → команда молча падала в on-error → мастер писал «создайте вручную» ровно на том железе, для которого делался. Исправлено: правильный синтаксис + `configuration.mode=ap`, фолбэк wpa2+wpa3 → wpa2, затем фолбэк на старый драйвер `/interface/wireless` (`mode=ap-bridge`, security-profile). Определение мастер-радио теперь устойчиво к обоим вариантам «пустого» master-interface (ошибка чтения, пустая строка, `none`).

**🔴 3. Policy-routing kill-switch — ПОДТВЕРЖДЁН БАГ.** Маршрут в пользовательской таблице, чей шлюз лежит в подсети из main, требует `gateway=172.18.20.6@main` (док. «Policy Routing», пример `gateway=172.16.2.1@main routing-table=myTable`). Без `@main` шлюз в таблице `vpn-rt` не резолвится → маршрут неактивен → **в VPN-подсети не было бы интернета даже при работающем туннеле**. Исправлено везде (включая текст подсказки «добавьте позже»).

**Сертификаты (новое, в хендоффе не было).** С ROS **7.21** параметр `builtin-trust-anchors` заменён на `builtin-trust-store` (changelog 7.21: “certificate - added option to configure built-in trust store (replaced "builtin-trust-anchors" parameter)”). Рабочая команда: `/certificate/settings/set builtin-trust-store=fetch`. Старый скрипт знал только `builtin-trust-anchors=trusted` → на свежих прошивках шаг падал в on-error с сообщением «ROS < 7.19», и загрузка образа могла падать по TLS. Исправлено: пробуем 7.21+ → 7.19–7.20 → предупреждение.

**device-mode.** `print` показывает `container: yes`, но `get` в скриптах может вернуть boolean. Старый .rsc сравнивал с `true`, старый ps1 — со строкой `yes`; оба варианта не могли быть верны одновременно. Исправлено: оба скрипта принимают `true|yes` (`[:tostr ...]` в .rsc, `-match "^(yes|true)$"` в ps1).

**🔴 2. PowerShell по SSH — найдено 5 проблем, все исправлены:**
- `ssh-keygen -N ""` — Windows PowerShell 5.1 «съедает» пустой аргумент → ssh-keygen интерактивно спрашивал passphrase и всё висло. Фикс: на PS < 6 передаётся `'""'`.
- `/system/reboot` и `/system/device-mode/update` в интерактивной сессии просят подтверждение/держат сессию → зависание. Фикс: обёрнуты в `:execute script="..."` (скриптовый контекст не спрашивает подтверждения, ssh возвращается сразу).
- Архитектура `x86_64` ошибочно отвергалась (guard был `arm/arm64/x86`), а URL Extra packages для неё должен быть `all_packages-x86-...zip`. Фикс: guard + маппинг `x86_64→x86`.
- `StrictHostKeyChecking=accept-new` не поддерживается старым OpenSSH 7.7 (Windows 10 1809) → заменено на `=no`.
- Старые Windows 10 + PS 5.1 не включают TLS 1.2 по умолчанию → `Invoke-WebRequest` к download.mikrotik.com падал. Фикс: принудительный TLS 1.2.
- Бонус: уборка теперь удаляет только ключи мастера (комментарий `mt-vless-wizard`, поля `info`/`key-owner` для новых/старых ROS), а не все SSH-ключи пользователя.

**Прочее подтверждено как корректное:** синтаксис `/container/envs` (`list=` сейчас, `name=` на старых — try/catch в силе), `/container/config username/password` (с 7.8), veth `gateway6` с фолбэком, импорт образа через tar (`/container/add file=`), registry-url (по умолчанию lscr.io — явное указание Docker Hub обязательно), батчинг маршрутов (основной путь — `/import` файлом).

## 2. Прочие исправленные баги генератора

- **Экранирование значений**: `$`, `"`, `\` в паролях/SSID/комментариях ломали RouterOS-строки (`$` начинает подстановку переменной). Теперь всё экранируется (`\$ \" \\`); IP/логин для ps1 фильтруются отдельно.
- **Кривые CIDR абортили импорт**: `1.2.3.4/24` вызывал ошибку `/ip/route/add`, а ошибка внутри `:foreach` останавливает весь /import на середине. Теперь CIDR нормализуются по границе сети ещё в браузере, а каждый add обёрнут в `:do on-error` с сообщением.
- **Маркировка локального трафика**: kill-switch-правило заворачивало в туннель и трафик к локальным сетям (роутер, принтеры, NAS). Добавлены accept-исключения для RFC1918 перед mark-правилом; порядок правил гарантирован (remove по комментариям → add в правильном порядке).
- **MSS-диапазон**: `1453-65535` оставлял дыру 1361–1452 (пакеты не клампились → подвисания). Теперь `1361-65535 → new-mss=1360`. Также исправлено в autosetup.rsc и памятке.
- **Режим «весь трафик»**: если адрес сервера — домен и `:resolve` упал, добавление маршрута-исключения роняло скрипт. Теперь обёрнуто с предупреждением о возможной петле.
- **Выбор USB-диска**: приоритет слотам с именем `usb*`, потом любой не-tmpfs (меньше шансов схватить не тот носитель).
- Мелочь: кнопка «Скопировать» восстанавливала не тот текст.

## 3. Переработка мастера (по запросу)

Новый флоу `vless-mikrotik-wizard.html`:
1. **Пролог** при открытии: что за файл, как работает, что делает с роутером (свёртка с деталями), два «физических» предусловия.
2. **Ключ VLESS/подписка** — как раньше (парсер, выбор из нескольких, предупреждения).
3. **Способ установки**: авто по SSH — **сначала `/system/backup` + `/export`, оба файла скачиваются на ПК**; при недоступности бэкапа установка останавливается (можно осознанно продолжить). Вручную — готовые строки бэкапа отдельным блоком + скрипт.
4. **Подсеть**: действующая сеть ↔ отдельная VPN-подсеть (bridge-vpn, DHCP, kill-switch с исключениями для локалки).
5. **Маршрутизация**: для действующей сети — список/весь трафик/нет + встроенный DomainMapper; для новой подсети — объяснение, что списки не нужны, и опциональный список для основной сети. Маршруты применяются только если контейнер поднялся.
6. **Wi-Fi**: оставить как есть ↔ новый SSID; если новый SSID при действующей подсети — мастер попросит параметры подсети (иначе трафик не отделить) и включит её автоматически.
7. **Отчёты**: на странице — сводка «что будет сделано»; в конце .rsc — «ИТОГОВЫЙ ОТЧЁТ» с реальными статусами (контейнер, число маршрутов, подсеть, Wi-Fi, как проверить, как удалить).
8. **Живые логи**: каждый шаг .rsc пишет «что и зачем» (`=== [n/N] … / Зачем: …`); setup.ps1 ведёт рассказ по стадиям и транслирует лог роутера построчно (`[router] …`).
9. **Адаптация прошивок**: wifi ↔ wireless, trust-store 7.21+ ↔ trust-anchors 7.19–7.20, envs `list=` ↔ `name=`, device-mode `true` ↔ `yes`, veth с/без `gateway6`. Скрипт идемпотентен — повторный запуск не плодит дубликатов.

## 8. Дополнение от того же дня: карточка роутера, адаптивность, macOS

По второму запросу Виталия добавлено:

**Карточка роутера с фото.** Оба установщика (setup.ps1 и setup.sh) после логина на роутер берут `board-name`, строят слаг страницы модели на mikrotik.com, вытаскивают из неё `og:image` (вариант `_lg.webp`) и открывают в браузере локальную карточку `router-info.html`: фото, модель, RouterOS, архитектура, свободное место, ссылка на страницу модели. Проверено живьём: «hAP ax^2» → slug `hap_ax2` → фото найдено (HTTP 200). Если модель не нашлась или нет интернета — карточка без фото, с моделью и ссылкой на поиск (graceful fallback, проверено на «hEX»). Установка при этом не блокируется.

**Адаптивность страницы.** Ширина теперь `clamp(680px, 78vw, 1280px)` — на больших мониторах страница шире, на ноутбуках компактнее, на телефонах — в одну колонку. Все шрифты переведены в rem; в правом верхнем углу панель **A− / A+** (80–140%, шаг 10%), выбор запоминается (localStorage, с try/catch для file://).

**macOS/Linux.** В авто-режиме теперь генерируются **оба** установщика: `setup.ps1` (Windows) и `setup.sh` (macOS/Linux, bash 3.2-совместимый: ssh/scp/curl/unzip из коробки, открытие карточки через `open`/`xdg-open`). Страница определяет ОС пользователя и подсказывает, какой файл его. Важные кодировочные нюансы: setup.ps1 скачивается **с BOM** (иначе Windows PowerShell 5.1 читает кириллицу как ANSI-кракозябры), setup.sh — **без BOM** (иначе bash спотыкается о shebang).

**Тесты дополнены:** 40 комбинаций + проверки ps1/sh (карточка, x86_64, `:execute`, бэкап, BOM/no-BOM), `bash -n` реального сгенерированного setup.sh — OK, живой тест определения фото. Не проверено на железе: то же, что в §5, плюс открытие router-info.html на реальных Windows/macOS.

## 9. Дополнение: автоматический фолбэк на локальный образ (.tar)

Запрос: если контейнер не качается с Docker Hub (типичная беда в РФ), после 3 попыток поставить его из локального файла автоматически.

**Логика в `.rsc` (на роутере).** Шаг «Контейнер» переписан: (1) если на роутере уже лежит образ `.tar` (по заданному имени или по маске `*vless*/*sing*/*sbox*/*container*.tar`) — сразу `/container/add file=` (Docker Hub вообще не трогается); (2) иначе — до **3 попыток** `remote-image` с паузами (каждая попытка сносит недокачанный контейнер и пробует заново, ждёт статус до 5 мин); (3) если все 3 провалились и файла нет — печатает точную инструкцию, как получить `.tar` (`podman pull --arch=… ; podman save …`, с правильным маппингом x86_64→amd64) и залить его.

**Каскад в установщиках (ps1/sh).** После `/import` установщик проверяет, есть ли running-контейнер. Если нет — добывает `.tar` по приоритету: **(а)** файл рядом со скриптом (`vless-container-<arch>.tar`, или имя из мастера) → **(б)** скачать по URL из мастера → **(в)** собрать через `podman`/`docker`, если они есть на ПК. Найденный tar заливается по scp на роутер, затем **повторный `/import`** — `.rsc` видит образ, ставит из файла, и за счёт идемпотентности до-настраивает маршруты/Wi-Fi/kill-switch в том же прогоне. Для нетехнической аудитории рабочий путь — вариант (а): автор один раз готовит `vless-container-arm64.tar`/`-arm.tar` и раздаёт подписчикам вместе с мастером.

**UI.** В шаге 6 блок «Запасной образ»: поле URL, поле имени файла, свёрнутая инструкция по подготовке `.tar` под обе архитектуры. Ничего заполнять не обязательно — по умолчанию ищется `vless-container-<arch>.tar` рядом со скриптом.

**Тесты:** проверки `.rsc` (tar-first ветка, цикл `:for 1..3`, подсказка podman) и ps1/sh (проверка running, каскад файл/URL/podman, повторный import) добавлены во все 40 комбинаций — зелёные; `bash -n` пересобранного setup.sh — OK; аудит кавычек ps1 — чисто (here-string карточки и regex в одинарных кавычках — не ошибки). Не проверено на железе: сам сценарий «Docker Hub недоступен → tar» (нужен роутер в РФ-сети или заблокированный registry).

## 10. Готовое зеркало образа + плейсхолдер {arch}

По запросу «скачать контейнер и положить на GitHub»: образ `wiktorbgu/vless-sing-box-tunnel-mikrotik:latest` выкачан из Docker Hub через `crane` (без docker/podman в песочнице) под три платформы и сохранён в docker-archive `.tar` — папка `container-images/`:

- `vless-container-arm64.tar` (17 МБ, для ax²/ax³ и новых), `vless-container-arm.tar` (16 МБ, armv7/старые), `vless-container-x86_64.tar` (18 МБ, x86/CHR);
- у всех корректный `RepoTags` в manifest.json (RouterOS по нему находит образ), контрольные суммы в `SHA256SUMS.txt`, это точная копия оригинала (тот же digest);
- `container-images/README.md` — инструкция: как выложить на GitHub Releases или raw и что вписать в мастер.

Установщик доработан под «один линк на все модели»: в поле «Ссылка на .tar» можно указать URL с плейсхолдером `{arch}` — ps1/sh подставят `arm64`/`arm`/`x86_64` под конкретный роутер. Поиск файла рядом со скриптом расширен синонимами архитектур (`arm`↔`armv7`, `x86_64`↔`amd64`↔`x86`), чтобы имена зеркал и RouterOS `architecture-name` всегда совпадали. Docker Hub остаётся первым источником (свежесть) — зеркало включается только после 3 неудачных попыток.
