# Отчёт о живом тесте на реальном роутере — 6 июля 2026

Тест проводился прямо на твоём роутере **hAP ax² (arm64), RouterOS 7.18.2**, подключённом за вторым NAT (WAN ether1 = 192.168.1.105 от вышестоящего роутера). Доступ по SSH. Перед любыми изменениями сделана резервная копия (`pretest-vless.backup`, лежит в Files роутера и в папке проекта). По твоей просьбе удалён нерабочий WireGuard `wg1` (получал 0 байт, rx=0) — его конфигурация сохранена в `wg1-saved.rsc` на роутере, вернуть можно в любой момент. После тестов роутер полностью очищен от тестовых артефактов и возвращён в рабочее состояние (интернет идёт через провайдера).

**Главный итог:** живой тест вскрыл **пять критических багов, которые симуляция поймать не могла в принципе** — это поведение самого парсера RouterOS во время выполнения. Все пять исправлены и проверены на железе. Плюс выяснилось, что скачанные ранее образы для GitHub были в нерабочем формате — пересобраны.

## Критические баги, найденные и исправленные на железе

### 1. Сертификаты роняли весь скрипт на RouterOS до 7.21

Команда `/certificate/settings/set builtin-trust-store=fetch` (и `builtin-trust-anchors` для 7.19–7.20) на прошивке, где такого параметра ещё нет, вызывает **ошибку парсинга**, а не выполнения. Ключевой нюанс RouterOS: обычная защита `:do { … } on-error={}` ловит только ошибки выполнения, а ошибки парсинга (неизвестный параметр) — **не ловит**, и весь скрипт падает целиком. На твоей 7.18.2 скрипт умирал на шаге сертификатов, ещё не дойдя до контейнера.

**Фикс:** команда теперь выполняется через `[:parse "…"]` — это единственный способ в RouterOS «отложить» компиляцию так, чтобы ошибку неизвестного параметра можно было поймать. Проверено: на 7.18.2 скрипт спокойно проходит мимо (сертификаты просто не настраиваются, что для 7.18 нормально).

### 2. Переменные окружения: на 7.18 параметр называется `name=`, а не `list=`

Это оказалось сюрпризом: в документации MikroTik примеры используют `list=`, но на 7.18.2 живьём работает только `name=` (параметр переименовали в более новых версиях). Наш код пробовал `list=` с откатом на `name=` через `on-error` — но это опять **ошибка парсинга**, которую `on-error` не ловит. Скрипт падал на первой же переменной.

**Фикс:** сначала определяем рабочий синтаксис пробой через `:parse` (list → name), затем все 8 переменных добавляем через `:parse` с подстановкой найденного имени. Проверено на железе: все 8 переменных из твоего ключа легли корректно (`name="vless"`).

### 3. Кавычки в тексте подсказок ломали парсинг

В одном из пояснений было написано `:put "…MSS-клампинг чинит "подвисающие" сайты…"` — внутренние кавычки вокруг слова рвали строку. Хитрость в том, что кавычек чётное число, поэтому мой прежний автоматический аудит их пропускал, а парсер RouterOS — падал.

**Фикс:** заменил на «ёлочки», и — важнее — заменил слабую проверку на честную эмуляцию лексера RouterOS. Больше таких мест в скрипте нет.

### 4. Wi-Fi: обращение к «чужому» драйверу роняло скрипт

Тут самое коварное. У MikroTik три разных меню Wi-Fi: `/interface/wifi` (новые, 7.13+), `/interface/wifiwave2` (ax на 7.9–7.12) и `/interface/wireless` (старый драйвер). На твоём ax² с драйвером wifi-qcom меню `/interface/wireless` **не существует**, и простое упоминание его в скрипте — даже в ветке, которая никогда не выполнится, — это ошибка парсинга, роняющая весь скрипт (RouterOS парсит весь файл, а не только исполняемые ветки). `:parse` эту конструкцию (цикл с поиском интерфейсов) не компилирует.

**Фикс (нетривиальный):** каждый из трёх драйверов оформлен как отдельный временный `/system/script` — его текст сохраняется без компиляции, а при запуске ошибка отсутствующего меню всплывает и штатно ловится `on-error`. Скрипт перебирает драйверы, пока один не сработает. Проверено на железе: на 7.18.2 создались **vpn-wifi1 и vpn-wifi2** с нужным SSID, паролем и WPA2/WPA3, поверх обоих радиодиапазонов.

### 5. Формат образа для GitHub был нерабочим

Образы, которые я подготовил ранее (через `crane` в режиме по умолчанию), RouterOS **не смог прочитать**: в логах контейнера `error getting layer file / import finished but no image found`. Причина — режим по умолчанию сжимает слои в gzip, а RouterOS ждёт несжатые слои (формат `docker save`). То есть весь запасной-образ-с-GitHub не заработал бы.

**Фикс:** пересобрал все три образа (`arm64`, `arm`, `x86_64`) в правильном формате (`crane --format=legacy`) — заменены в папке `container-images/`. Проверено на железе: образ в новом формате **импортируется успешно** (`import successful`), контейнер создаётся.

## Что подтверждено работающим на реальном железе

- Установка контейнера из локального `.tar` (минуя Docker Hub, который с твоего РФ-роутера **действительно недоступен** — проверено: `registry-1.docker.io` не отвечает по HTTPS). Это ровно тот случай, ради которого делался запасной образ.
- Все 8 переменных окружения из ключа применяются.
- veth-интерфейс, firewall (MSS + NAT), отдельная VPN-подсеть (bridge + DHCP-пул), kill-switch (таблица маршрутизации + правила mangle) — создаются корректно.
- Wi-Fi: отдельный SSID на обоих радио.
- Маршрутизация по списку: 3 подсети добавились с правильным шлюзом (172.18.20.6) и комментарием, батчинг работает.
- **Идемпотентность:** повторный запуск не плодит дубликаты (маршрутов осталось 3, а не 6; Wi-Fi-интерфейсов 2, а не 4).

## Что НЕ заработало и почему это не про наш скрипт

**Твой ключ `govless` — это VLESS+TLS (не Reality).** А образ `wiktorbgu/vless-sing-box-tunnel-mikrotik` жёстко зашит под **Reality** (в его конфиге `reality.enabled: true` не отключается). С TLS-ключом контейнер падает: `FATAL: invalid public_key`. Это не ошибка скрипта — мастер как раз предупреждает, когда в ключе нет `pbk`. Для этого образа нужен **Reality-ключ** (с `pbk` и `sid`), либо другой образ/ядро (например, mihomo из инструкции №32), которое умеет TLS.

Ради интереса я собрал патченный образ (сделал Reality необязательным + добавил ALPN + подставил IP сервера вместо домена, чтобы обойти петлю DNS внутри туннеля) — и sing-box с твоим ключом **стартовал без ошибок**. Твой сервер `myphotostudio.site` живой и отвечает (валидный TLS-сертификат, пинг 42 мс). Но доводить сквозной проход трафика через этот самодельный образ я не стал — это уже не про наш инструмент, а про совместимость ключа с конкретным контейнером.

## Отдельная находка: место на ax² без USB

Распакованный образ занимает **50 МБ**, а на твоём ax² изначально было свободно только 43 МБ — образ физически не влезал. Причина — на флешке лежал **остаток старого контейнера mihomo** (от инструкции №32), занимавший ~52 МБ. После его удаления стало 93 МБ, и всё встало штатно. Вывод для инструкции: на ax² без USB место впритык; стоит либо ставить контейнер на USB, либо чистить старые образы, либо использовать распаковку в RAM. Порог предупреждения в мастере поднят.

## Состояние роутера сейчас

Полностью очищен от тестовых артефактов. В Files остались только: твой бэкап `pretest-vless.backup`, сохранённый конфиг `wg1-saved.rsc` (для возврата WireGuard) и твой `peer-1.conf`. device-mode для контейнеров как был (`yes`). Интернет идёт через провайдера. Если захочешь вернуть `wg1` — скажи, восстановлю из сохранённого конфига за минуту.
