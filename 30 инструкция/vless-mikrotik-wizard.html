ℹ

Что это и как работает

Этот файл — автономный мастер настройки. Он работает целиком у вас в браузере: ничего никуда не отправляет (в интернет ходит только за списками доменов с GitHub и за DNS-резолвингом, если вы сами это попросите на шаге «Маршрутизация»). Пароли от роутера и Docker Hub страницу не покидают.

Как устроен процесс:

  1. Вы отвечаете на вопросы шагов 1–6 — мастер собирает под них скрипт настройки vless-setup.rsc.
  2. Авто-режим: дополнительно генерируется установщик под вашу систему — setup.ps1 (Windows) или setup.sh (macOS/Linux). Он сам зайдёт на роутер по SSH, определит модель и покажет её карточку с фотографией, сделает резервную копию и скачает её на ПК, проверит все зависимости (версию RouterOS, архитектуру, пакет container, режим контейнеров, свободное место) и, только если всё в порядке, выполнит установку. Любая проблема — установка останавливается с объяснением.
  3. Ручной режим: вы получите готовые строки: сначала команды резервной копии, потом сам скрипт — вставить в терминал или выполнить через /import. Все проверки зависимостей встроены в начало скрипта — при проблеме он остановится и подскажет, что исправить.
  4. Во время работы скрипт ведёт «прямой эфир»: пишет в консоль, что он сейчас делает и зачем это нужно.
  5. В конце — отчёт: что настроено, что проверить, что делать, если что-то не так.
Что именно скрипт делает с роутером

Создаёт veth-интерфейс 172.18.20.5/30 ↔ 172.18.20.6 (виртуальный кабель к контейнеру), правила firewall (MSS-клампинг, чтобы сайты не «подвисали», и NAT в туннель), переменные окружения из вашего ключа vless://, скачивает и запускает контейнер wiktorbgu/vless-sing-box-tunnel-mikrotik. Дальше — по вашему выбору: маршруты по списку, отдельная VPN-подсеть с DHCP и защитой от утечек (kill-switch), отдельный Wi-Fi SSID. Всё создаваемое помечено комментариями vless-… — легко найти и удалить. Скрипт можно запускать повторно: он не плодит дубликаты.
Если Docker Hub недоступен (частая проблема в РФ), скрипт делает 3 попытки скачать контейнер, а затем ставит его из готового файла .tar — см. «Запасной образ» на шаге 6.

Два действия нельзя сделать скриптом издалека (ограничение MikroTik, «физика»):
1. Установка пакета container-<версия ROS>-<arm/arm64>.npk — авто-режим скачает и установит его сам, вам останется дождаться перезагрузки. hAP ax² — это arm64.
2. /system/device-mode/update container=yes подтверждается только выключением питания из розетки (или кнопкой reset) — программный reboot не считается. Авто-режим попросит вас один раз дёрнуть питание в нужный момент.
Скрипт проверяет оба пункта и не продолжает, пока они не выполнены.

Способ установки

Пароль в файл не сохраняется — его запросит консоль при первых 1–2 подключениях (безопаснее), дальше скрипт работает по временному SSH-ключу и сам удалит его в конце. Windows 10/11 — встроенный OpenSSH; macOS и Linux — ssh есть из коробки.

3

Как будем использовать VPN

Шлюзом будет адрес .1, DHCP раздаст .10–.254. Подсеть не должна пересекаться с основной (обычно 192.168.88.0/24).

4

Маршрутизация

Вы создаёте отдельную VPN-подсеть — весь её трафик и так пойдёт через туннель, списки для неё не нужны.

Вставьте подсети — подойдёт вывод DomainMapper (raw/cidr или mikrotik-route), просто строки CIDR или вообще любой текст: выну все адреса вида x.x.x.x/nn сам. Адреса с «хвостом» (1.2.3.4/24) выровняю по границе сети автоматически.

🔮 Собрать список автоматически (встроенный DomainMapper)

Отметьте сервисы — списки доменов возьму из репозитория DomainMapper (Ground-Zerro), разрезолвлю через DNS-over-HTTPS прямо в браузере и сложу в подсети. Большие списки (Antifilter, ITDog — тысячи доменов) собираются несколько минут.

Загружаю список сервисов…

Добавьте свои домены — по одному на строку (можно с *., можно готовые CIDR). Они добавятся к выбранным сервисам. Или загрузите файлом.

5

Wi-Fi

Отдельный Wi-Fi технически требует отдельной подсети (иначе его трафик не отделить от основной сети) — она будет создана автоматически с kill-switch.

Запасной образ (если Docker Hub недоступен из вашей сети)

Скрипт сначала до 3 раз пробует скачать контейнер с Docker Hub. Если не вышло — поставит его из готового файла .tar, интернет к Docker больше не нужен. Образ ищется по порядку: файл рядом со скриптом → скачивание по ссылке → сборка через podman/docker на ПК (если установлены). Ничего заполнять не обязательно — по умолчанию скрипт ищет файл vless-container-<архитектура>.tar рядом с собой.

Резерв уже настроен: в поле выше — готовая ссылка на образ на GitHub автора (с плейсхолдером {arch} — установщик сам подставит arm64, arm или x86_64). Если Docker Hub недоступен, образ скачается отсюда автоматически. Можно заменить на свою ссылку или очистить.

Как подготовить .tar (один раз, на ПК с podman/docker или на зарубежном VPS) 
# arm64 — hAP ax², ax³ и большинство новых
podman pull --arch=arm64 docker.io/wiktorbgu/vless-sing-box-tunnel-mikrotik:latest
podman save docker.io/wiktorbgu/vless-sing-box-tunnel-mikrotik:latest -o vless-container-arm64.tar

# arm (32-бит, старые устройства)
podman pull --arch=arm docker.io/wiktorbgu/vless-sing-box-tunnel-mikrotik:latest
podman save docker.io/wiktorbgu/vless-sing-box-tunnel-mikrotik:latest -o vless-container-arm.tar

Положите нужный файл в ту же папку, что и setup.ps1/setup.sh (или раздайте его подписчикам). Docker: docker pull --platform linux/arm64 … и docker save … -o ….
